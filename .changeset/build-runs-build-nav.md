fix(devkit): reliable navigation codegen across build + dev, and a non-hanging desktop build

The navigation layer is generated by `build:nav` into four files
(`navigation.native.tsx`, `navigation.tsx`, `routes.d.ts`, `index.ts`) that are
gitignored as build artifacts. Several commands assumed those files already
existed, so a clean checkout (clone, CI, teammate, fresh machine) failed — e.g.
Metro: `Unable to resolve module ./navigation.js from src/navigation/index.ts`.

- `sublime build` (mobile) now runs `build:nav` before prebuild/Gradle and aborts
  with a clear message on storybook validation errors.
- New `sublime dev:web` runs Vite with **live navigation recompilation** —
  compiles once, then watches the storybooks and rebuilds on change, tearing the
  watcher down when Vite exits. Scaffolded `dev:web` uses it.
- `sublime dev:desktop` (renamed from `desktop:dev`; old name kept as an alias)
  now does the same live nav recompilation around Electron Forge.
- `sublime build:desktop` is the new name for `desktop:build` (old name aliased).
- Desktop `make` no longer hangs on Windows: the default maker is now **ZIP on
  every OS** (fast, portable, no installer-packing hang). The Windows Squirrel
  maker is opt-in (documented in `forge.config.ts`); `maker-squirrel` is no
  longer a scaffolded dependency. Deb/Rpm still cover Linux.
- `build:nav --watch` (and the new watchers) no longer crash on web-only or
  native-only projects (only existing storybooks are watched).
- Scaffold-time `build:nav` failures are surfaced instead of silently swallowed.
